# Tech-Spec: Host Agent Config Inheritance

**Created:** 2026-03-25

## Overview

### Problem Statement

Users authenticating via OAuth/SSO (e.g., `claude login`, Gemini auth) who cannot create standalone API keys have no way to authenticate the sandboxed agent without manually re-authenticating inside the container. The sandbox currently only supports injecting credentials via environment variables (`secrets`), which requires an API key. OAuth-based auth stores tokens in the host's agent config directory (`~/.claude`, `~/.config/gemini`), and the sandbox has no mechanism to make these available to the containerized agent.

### Solution

Add a `host_agent_config` configuration field that mounts the host's agent config directory read-write into the container at the correct path for the sandbox user. The field accepts explicit `source` and `target` paths with well-documented defaults per agent. Tilde (`~`) expansion is added to mount path resolution so `~/.claude` resolves correctly.

### Scope

**In Scope:**
- New `host_agent_config` config field with explicit `source`/`target` and documented defaults per agent
- Tilde (`~`) expansion in mount source path resolution in `sandbox.sh`
- Read-write bind mount of host agent config dir to sandbox user home
- Starter config template update with commented example and defaults
- Config parsing for the new field
- Tests for tilde expansion, config parsing, and mount flag assembly

**Out of Scope:**
- Generic read-only mount support for arbitrary mounts entries
- Changes to `Dockerfile.template`
- Token refresh propagation or auth monitoring
- Automated re-authentication flows

## Context for Development

### Codebase Patterns

- **Config parsing**: All fields extracted via `yq eval` with null-coalescing (`// ""`) in `parse_config()` at `sandbox.sh:91-179`. New fields follow the same pattern. Optional fields default to empty string, checked with `[[ -z ... ]]`.
- **Mount storage**: Sources and targets stored in parallel bash arrays `CFG_MOUNT_SOURCES` / `CFG_MOUNT_TARGETS` (sandbox.sh:79-80). Iterated with `${!CFG_MOUNT_SOURCES[@]}` index loop.
- **Mount resolution**: In `cmd_run()` at `sandbox.sh:485-495`. Currently a two-way branch: absolute paths pass through, relative paths resolve via `cd "${config_dir}/${src}" && pwd`. Tilde expansion adds a third case before the relative check.
- **Entrypoint user switch**: `entrypoint.sh:5-16` runs as root first (fixes mount propagation, unmounts /proc masks), then drops to sandbox user via `runuser -u sandbox`. UID alignment logic slots into the root block before the `runuser` call.
- **Docker flag assembly**: `run_flags` bash array in `cmd_run()` (sandbox.sh:455-503). Mounts added as `-v "${src}:${tgt}"`. Working directory set to first mount target.
- **Template config**: `templates/config.yaml` uses `#` comments with example values. Optional sections shown commented out.
- **Test patterns**: TAP-like runner in `tests/test_sandbox.sh`. Each test: create `tmpdir` + `mockbin`, call `setup_build_mock`, write inline config YAML, run sandbox capturing output, assert on `docker.log` contents, cleanup. Tests grouped under `# AC:` section headers.

### Technical Decisions

1. **Read-write mount, not copy-on-start** — OAuth providers typically use rotating refresh tokens. A copy-on-start approach would cause the sandbox's token refresh to invalidate the host's refresh token, breaking host authentication after long sessions. Read-write mount keeps tokens synchronized.

2. **Explicit paths with documented defaults** — `host_agent_config` takes `source` and `target` fields rather than a boolean. Defaults are documented per agent (`~/.claude` for claude-code, `~/.config/gemini` for gemini-cli) but users can override for non-standard setups.

3. **Tilde expansion in all mount sources** — not scoped to `host_agent_config` only. The `~` prefix is expanded to `$HOME` in the existing mount resolution loop, benefiting both regular `mounts` and `host_agent_config`.

4. **Gemini config path needs verification** — `~/.config/gemini` is a placeholder. The exact path must be confirmed before implementation. The spec uses this as the default but flags it as TBD.

5. **`host_agent_config` mount appended after regular mounts** — the config mount is added to `run_flags` separately from the `mounts` loop, so it doesn't affect the `-w` working directory logic (which uses the first regular mount's target).

6. **Source directory existence validated at runtime** — `cmd_run()` checks that the `host_agent_config.source` directory exists before adding the `-v` flag, and `die`s with a clear error if it doesn't.

7. **UID alignment for cross-platform support** — On Linux, Docker bind mounts preserve host UID/GID. The container's `sandbox` user (UID 1000) may not match the host user's UID, making the mounted config directory unreadable. When `host_agent_config` is configured, `sandbox.sh` passes `HOST_UID` and `HOST_GID` env vars, and `entrypoint.sh` adjusts the sandbox user's UID/GID to match before dropping privileges. On macOS (Docker Desktop), VirtioFS remaps UIDs transparently, so the alignment is a harmless no-op.

  - File: `sandbox.sh` (after the `mcp` parsing block, ~line 178)
  - Action: Add parsing for the new `host_agent_config` field:
    ```bash
    # Extract host_agent_config (optional object with source/target)
    CFG_HOST_AGENT_CONFIG_SOURCE="$(yq eval '.host_agent_config.source // ""' "${CONFIG_PATH}")"
    if [[ "${CFG_HOST_AGENT_CONFIG_SOURCE}" == "null" ]]; then CFG_HOST_AGENT_CONFIG_SOURCE=""; fi
    CFG_HOST_AGENT_CONFIG_TARGET="$(yq eval '.host_agent_config.target // ""' "${CONFIG_PATH}")"
    if [[ "${CFG_HOST_AGENT_CONFIG_TARGET}" == "null" ]]; then CFG_HOST_AGENT_CONFIG_TARGET=""; fi
    ```
  - Notes: Both fields are optional. If `host_agent_config` key is absent, both resolve to empty string. If only `source` is provided without `target` (or vice versa), the validation in Task 4 handles it.

- [x] **Task 4: Add `host_agent_config` mount to `cmd_run()` flag assembly**
  - File: `sandbox.sh` (after the regular mounts loop and `-w` flag, ~line 501)
  - Action: Add host agent config mount logic:
    ```bash
    # Mount host agent config directory (if configured)
    if [[ -n "${CFG_HOST_AGENT_CONFIG_SOURCE}" || -n "${CFG_HOST_AGENT_CONFIG_TARGET}" ]]; then
      if [[ -z "${CFG_HOST_AGENT_CONFIG_SOURCE}" || -z "${CFG_HOST_AGENT_CONFIG_TARGET}" ]]; then
        die "host_agent_config requires both source and target" 1
      fi
      local hac_src="${CFG_HOST_AGENT_CONFIG_SOURCE}"
      # Apply tilde expansion
      if [[ "${hac_src}" == "~/"* ]]; then
        hac_src="${HOME}/${hac_src#\~/}"
      elif [[ "${hac_src}" != /* ]]; then
        hac_src="$(cd "${config_dir}/${hac_src}" && pwd)"
      fi
      if [[ ! -d "${hac_src}" ]]; then
        die "host_agent_config source directory not found: ${hac_src}" 1
      fi
      run_flags+=("-v" "${hac_src}:${CFG_HOST_AGENT_CONFIG_TARGET}")
      # Pass host UID/GID for cross-platform permission alignment
      run_flags+=("-e" "HOST_UID=$(id -u)" "-e" "HOST_GID=$(id -g)")
    fi
    ```
  - Notes: Placed AFTER the `-w` flag logic so it doesn't affect working directory. Validates both fields present. Validates source directory exists. Applies same tilde expansion as regular mounts. Passes `HOST_UID`/`HOST_GID` so `entrypoint.sh` can align the sandbox user's UID.

- [x] **Task 4.5: Add UID alignment to `entrypoint.sh`**
  - File: `scripts/entrypoint.sh` (in the root block, before `exec runuser -u sandbox`, ~line 14)
  - Action: Add UID/GID alignment when `HOST_UID` is set and differs from the sandbox user's current UID:
    ```bash
    # Align sandbox user UID/GID with host user for bind mount permissions
    if [[ -n "${HOST_UID:-}" ]]; then
      current_uid="$(id -u sandbox)"
      if [[ "${HOST_UID}" != "${current_uid}" ]]; then
        usermod -u "${HOST_UID}" sandbox
        groupmod -g "${HOST_GID:-$(id -g sandbox)}" sandbox
        chown -R sandbox:sandbox /home/sandbox
      fi
    fi
    ```
  - Notes: Runs in the root block before `runuser` drops privileges. On macOS with Docker Desktop, VirtioFS handles UID mapping transparently, so even if the UIDs already match or the alignment is redundant, it's harmless. The `chown -R /home/sandbox` ensures any pre-existing files in the sandbox home (created during image build) are updated to the new UID.
